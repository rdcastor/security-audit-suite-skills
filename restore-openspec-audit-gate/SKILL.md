Re-apply the security-audit gate to the OpenSpec workflow skills after they've been regenerated by the `openspec update` CLI.

## When to use

- The user ran `openspec update` (or `openspec init`) and reported the audit step disappeared.
- A fresh checkout or `openspec update --force` was run.
- The user invokes this skill by name.

## What gets patched

Two files, each also mirrored to `.gemini/skills/...` and `.codex/skills/...`:

1. `.claude/skills/openspec-apply-change/SKILL.md` — adds a mandatory security-audit step before archive suggestion.
2. `.claude/skills/openspec-archive-change/SKILL.md` — adds a mandatory security-audit gate before delta sync / archival.

## Steps

1. **Check each target file** for the marker string `security-audit` (case-sensitive). If present in both Claude files, announce "Audit gate already present" and stop — nothing to do.

2. **Read** `.claude/skills/openspec-apply-change/SKILL.md` to confirm it matches the expected upstream structure (step 7 = "On completion or pause, show status").

3. **Patch `openspec-apply-change` (Claude)** — use `Edit`:

   - **old_string** (exact match):
     ```
     7. **On completion or pause, show status**

        Display:
        - Tasks completed this session
        - Overall progress: "N/M tasks complete"
        - If all done: suggest archive
        - If paused: explain why and wait for guidance
     ```
   - **new_string**:
     ```
     7. **On completion, run the security audit (mandatory before archive)**

        When all tasks are complete, invoke the `security-audit` skill scoped to this change **before** suggesting archive:

        ```
        Skill(skill="security-audit", args="<change-name>")
        ```

        - **BLOCK verdict**: Do not suggest archive. Surface findings, add remediation tasks to `tasks.md` (or re-open affected tasks), and pause for the user.
        - **PROCEED WITH FIXES verdict**: Surface Medium/Low findings and ask whether to fix now or defer; only suggest archive after user acknowledges.
        - **SAFE TO PROCEED verdict**: Include "✓ Security audit passed" in the completion output and suggest archive.

     8. **On completion or pause, show status**

        Display:
        - Tasks completed this session
        - Overall progress: "N/M tasks complete"
        - Security audit verdict (if all tasks complete)
        - If all done and audit passed: suggest archive
        - If audit blocked: explain why and wait for guidance
        - If paused: explain why and wait for guidance
     ```

   - Then `Edit` the Guardrails bullet list: append `- **Always run \`security-audit\` after the final task is marked complete and before suggesting archive** — never skip this step` after the `Use contextFiles from CLI output...` line.

4. **Patch `openspec-archive-change` (Claude)** — four edits:

   **(a)** Insert the audit gate before step 4:
   - **old_string**: `4. **Assess delta spec sync state**`
   - **new_string**: the full step-4 audit-gate block followed by `5. **Assess delta spec sync state**` (renumbered). See the live file in `.claude/skills/openspec-archive-change/SKILL.md` for the canonical block — if you need to reconstruct it, copy the step-4 text this skill produced on its last run.

   **(b)** Renumber: `5. **Perform the archive**` → `6. **Perform the archive**`

   **(c)** Renumber + augment:
   - **old_string**: `6. **Display summary**\n\n   Show archive completion summary including:\n   - Change name\n   - Schema that was used\n   - Archive location\n   - Whether specs were synced (if applicable)\n   - Note about any warnings (incomplete artifacts/tasks)`
   - **new_string**: same block renumbered to `7.`, with the line `- Security audit verdict (SAFE / PROCEED WITH FIXES / BLOCK-overridden)` inserted before the specs-sync line.

   **(d)** Insert `**Security audit:** ✓ SAFE TO PROCEED ...` line into the "Output On Success" template, and add the "Always run `security-audit` before archiving" bullet to **Guardrails**.

   If any `old_string` fails to match because upstream rewrote the skill, **stop and report the divergence** rather than guessing. Do not force the patch.

5. **Mirror to gemini + codex**:

   ```bash
   cp .claude/skills/openspec-apply-change/SKILL.md   .gemini/skills/openspec-apply-change/SKILL.md
   cp .claude/skills/openspec-apply-change/SKILL.md   .codex/skills/openspec-apply-change/SKILL.md
   cp .claude/skills/openspec-archive-change/SKILL.md .gemini/skills/openspec-archive-change/SKILL.md
   cp .claude/skills/openspec-archive-change/SKILL.md .codex/skills/openspec-archive-change/SKILL.md
   ```

6. **Verify** with `grep -l "security-audit" .claude/skills/openspec-apply-change/SKILL.md .claude/skills/openspec-archive-change/SKILL.md` — both should match. Report success.

## Guardrails

- **Idempotent**: if `security-audit` is already present in both target files, exit cleanly.
- **Fail loud on divergence**: if upstream changed the step numbering or wording such that the `old_string` no longer matches, stop and ask the user to reconcile. Don't partial-patch.
- **Do not edit `security-audit/SKILL.md` itself** — the openspec CLI doesn't touch it, so it doesn't need repair.
- **Do not commit**. Changes live in the working tree until the user commits.

## Why this exists

`openspec update` regenerates the OpenSpec skill files from the CLI's internal templates. Our audit gate is an intentional local patch on top of those templates. Rather than forking the skill names (which would also disconnect us from upstream fixes), we accept that the files are ephemeral and re-apply the patch via this skill. The authoritative backup of the gate lives in `CLAUDE.md`, which openspec does not touch.
